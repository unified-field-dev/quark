# Quark Registry Performance Study

**Performance study** ŌĆö standalone document. Baselines recorded 2026-06-01 on lab hardware. For reproducing benchmarks, see [`profiling.md`](profiling.md) and [`benches/compile-scale/README.md`](../benches/compile-scale/README.md).

> **Summary for adopters:** runtime registry lookups stay flat (~15ŌĆō21 ns) through 10k entries. Compile time grows roughly linearly with `inventory::submit!` count (~8 s lib compile at 10k submits on lab hardware). Keep registration counts per crate modest, isolate registrations into dedicated crates, and call `auto_discover()` once at startup.

---

## Abstract

Quark is a Rust library for type-safe, link-time registries backed by [`inventory`](https://docs.rs/inventory). This study evaluates four cost dimensions: runtime HashMap behavior after descriptors exist, cold compile cost of `inventory::submit!` macro expansion, incremental recompile when adding registrations, and heap memory at scale.

We present baseline measurements on WSL2 lab hardware (Intel i7-11700KF) at submit tiers from 1 to 10,000. Runtime lookups remain O(1) in practice through 10k entries. Compile cost scales roughly linearly with submit count. Incremental compile recompiles the whole registration crate when adding one submit. Integration tests (S1ŌĆōS4, T1ŌĆōT10) guard correctness and concurrency at scale.

---

## Research questions

1. Does lookup latency stay flat as registry size grows?
2. How does cold compile scale with `inventory::submit!` count?
3. What is incremental cost of adding one registration?
4. What is heap cost of building a registry at scale?

---

## Key findings

- **Runtime lookups:** `get_hit` / `get_miss` stay ~12ŌĆō21 ns regardless of registry size (1ŌĆō10k entries). HashMap lookup dominates; no linear scan.
- **Registry build:** `register_build` at 10k entries takes ~2.8 ms median ŌĆö acceptable for one-time startup discovery.
- **Cold compile:** lib compile grows from ~0.3 s (1 submit) to ~7.6 s (10k submits). Bin link stays cheap except at large tiers under lock contention.
- **Incremental compile:** adding one `submit!` recompiles the entire registration crate (~0.24ŌĆō0.52 s at 100ŌĆō1000 tier). Comment-only touches are slightly faster than cold.
- **Memory:** descriptor storage is static/leaked; registry map holds N string keys on heap. `inventory` link section grows with submit count (measured via compile-scale crates).
- **Quark does not scan crates at runtime** ŌĆö the linker merges registration sections from consumer rlibs at link time.

| Tier | Submits | Cold lib (s) | Cold bin link (s) | Notes |
|------|--------:|-------------:|------------------:|-------|
| 1 | 1 | 0.31 | 0.02 | |
| 10 | 10 | 0.25 | 0.03 | |
| 100 | 100 | 0.28 | 0.04 | |
| 500 | 500 | 0.52 | 0.02 | medium tier |
| 1000 | 1,000 | 0.80 | 0.02 | |
| 5000 | 5,000 | 3.44 | 2.57 | large tier |
| 10000 | 10,000 | 7.61 | 0.03 | lib compile dominates |

Compile cost scales roughly linearly with submit count at these tiers; 5000 bin link was slower due to build-directory lock contention after a large lib compile.

## Incremental compile (N = 100 and 1000)

Wall-clock seconds (`/usr/bin/time`) on `quark-scale-N` after prior build artifacts exist in `target/`.

| Tier | Cold lib (s) | +1 `submit!` (s) | Touch comment (s) | Bin link only (s) |
|------|-------------:|-----------------:|------------------:|------------------:|
| 100 | 0.33 | 0.24 | 0.21 | 0.02 |
| 1000 | 1.15 | 0.52 | 0.50 | 0.02 |

Adding one registration recompiles the whole crate; incremental savings appear on comment-only touches (still recompile, but faster than cold). Bin link stays cheap when the lib rlib is unchanged.

## Memory (`dhat-heap`)

```bash
cargo test --features dhat-heap memory_registry_at_scale -- --nocapture
```

At 10k manual `register()` via `Box::leak`, **dhat** reported **0 curr_bytes** at peak ŌĆö leaked allocations are not tracked as live heap in the same way as ordinary frees. Treat registry heap as:

| Component | Model |
|-----------|-------|
| Descriptors | `&'static T` in `.rodata` / leaked box storage |
| Registry map | `HashMap<String, &'static T>` ŌĆö string keys on heap (~N entries) |
| `inventory` link section | Grows with submit count; measured via compile-scale crates |

For heap profiling of non-leaked registries, use owned test descriptors or external tools (`massif`, `heaptrack`).

## Adoption guidance

When registration counts grow, consider these patterns:

1. **Registration isolation** ŌĆö move `inventory::submit!` entries into thin dedicated crates (e.g. `my-app-registrations`) so main application crates recompile less often.
2. **Fewer link-time submits** ŌĆö use lazy or explicit registration APIs for hot paths that do not need link-time discovery.
3. **Startup merge cache** ŌĆö call `auto_discover()` once during application bootstrap and pass owned registries through your app; avoid rebuilding on every request.

### Runtime vs compile-time

- **Runtime benches** measure `Registry<T>` HashMap behavior after descriptors exist.
- **Compile-scale crates** measure macro expansion + LLVM cost of `inventory::submit!` per crate.
